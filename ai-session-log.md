## Session Summary

Used Claude as the primary AI coding assistant throughout this assignment. The session covered:

### Phase 1 — Architecture Decision

**Prompt**: "I am applying for a Forward Deployed Engineer role. They want a Graph-Based Data Modeling & Query System over ERP business data. My stack: MERN. Deadline tonight. Give me: full system architecture decision (DB choice with tradeoffs), folder structure, graph data model, LLM prompting strategy."

**Claude's key decisions and reasoning**:

1. **PostgreSQL over MongoDB**: Data is fundamentally relational (Orders → Deliveries → Invoices → Payments). MongoDB would require application-level joins or duplication. LLMs generate accurate SQL; Mongo aggregation pipelines are less reliable for NL generation.

2. **PostgreSQL over Neo4j**: Graph traversal depth is shallow (≤5 hops). Neo4j free tier is limited. The "graph" is built at runtime as an in-memory adjacency structure from DB queries — best of both worlds.

3. **react-force-graph-2d over vis-network/D3**: vis-network looks dated. D3 requires 200+ lines of physics simulation code. react-force-graph-2d wraps D3's force simulation with a clean React API and handles node rendering efficiently.

4. **Groq over OpenAI/Gemini**: Free tier with ~300 tokens/sec. llama-3.3-70b-versatile is accurate enough for SQL generation.

5. **3-stage LLM pipeline**:
   - Stage 1: Zero-temperature binary guardrail classifier (5 tokens)
   - Stage 2: NL→SQL with full schema injection
   - Stage 3: Answer synthesizer grounded in result rows

### Phase 2 — Graph Data Model
